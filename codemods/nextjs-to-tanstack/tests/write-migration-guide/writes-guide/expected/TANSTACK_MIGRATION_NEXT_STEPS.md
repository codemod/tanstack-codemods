## Optional: i18n (TanStack Router `/{-$locale}/…`)

When this package had Next.js `i18n` (often via `next-i18next.config.*`), routes were moved under **`src/app/{-$locale}/**` and `createFileRoute` IDs use the optional segment **`/{-$locale}`** so the same route matches both the default locale (no prefix) and prefixed locales — see [Internationalization (i18n)](https://tanstack.com/router/latest/docs/guide/internationalization-i18n).

- If **`.codemod/i18n.json`** exists, it lists the locales the codemod detected. The workflow also emits **`src/i18n.ts`** (when absent) and wraps **`__root.tsx`** with **`I18nextProvider`**, and preloads **`import "./i18n"`** from **`router.tsx`** so **`react-i18next`** receives a real **`i18next`** instance (replacing **`next-i18next`** / `serverSideTranslations`).
- Wire **`react-i18next`** / **Paraglide** to the active locale (often `Route.useParams().locale` or router `rewrite` for URL localization — see the same doc and the [Paraglide + TanStack](https://github.com/TanStack/router/tree/main/examples/react/i18n-paraglide) example).
- Internal **`Link to="…"`** paths may need `params` updates so the optional `locale` is preserved when switching languages.
- **Strict TypeScript** — `Link to={...}` literals often break after routes move under `/{-$locale}/…`: internal paths must match the generated route union (`params`, `from`/validated `to`), or temporarily wrap/replace `Link` until every href matches the router tree (`vite dev`/build regenerates types).

## Next.js → TanStack Router API mapping

This mirrors the imperative navigation guide (**R4**, **R4b**, etc.):

| Next.js | TanStack Router |
| --- | --- |
| `next/navigation` | `useNavigate`, `useRouter`, `useLocation`, `useSearch`, loaders + `throw redirect()` |
| `next/router` (Pages Router) + `next/compat/router` | Same mappings when patterns are supported; skipped if `.prefetch`, `.events`, `.query`, or unsupported inline `useRouter()` members |
| `router.refresh()` | `router.invalidate()` (R4b dual-hook rewrite) |
| `notFound()` | `throw notFound()` from Router |
| Middleware / edge (`middleware.ts`) | `beforeLoad`, router context, or an external backend / proxy |
| `NEXT_PUBLIC_*` | Prefer `VITE_*` and `import.meta.env.*`; R13 adds `vite/client` types |
| `next/types`, `NextPage`, `GetServerSideProps`, … | `loader` + `Route.useParams` / `useSearch` / `useLoaderData`; `FileRoutesByPath` for explicit types — see **Replacing `next/types`** below |

### Replacing `next/types` and Next.js route generics

Next.js types describe **what the framework injects** (`params`, `searchParams`, `pageProps`). You import `NextPage`, `PageProps`, `GetServerSideProps`, etc., and thread generics through components.

TanStack Router **infers** from your `createFileRoute` definition: write a `loader`, optional `validateSearch`, then read data with `Route.useLoaderData()`, `Route.useParams()`, and `Route.useSearch()`. The route component usually takes **no props**.

**By pattern**

1. **Page / layout components** — Remove `NextPage`, `NextComponentType`, `PageProps` (`next/types`), and props like `{ params, searchParams }`. Use hooks on the file route’s `Route` object instead.
2. **Custom App (`AppProps`)** — Replace with `createRootRouteWithContext<Ctx>()` and pass `context` into `createRouter` so `beforeLoad` / `loader` see the same shape — [router context](https://tanstack.com/router/latest/docs/framework/react/guide/router-context).
3. **`getServerSideProps` / `getStaticProps` / `InferGet*PropsType`** — One `loader` replaces both; return the data you used to put in `props`. Use `throw notFound()` instead of `{ notFound: true }`.
4. **`getStaticPaths`** — No direct equivalent; prerender specific URLs via TanStack Start’s prerender configuration, or resolve unknown params at runtime in the `loader`.
5. **`NextApiRequest` / `NextApiResponse` / `NextApiHandler`** — Move to server routes, `createServerFn`, or an HTTP framework such as **Hono** (`Context`, `c.json()`).
6. **`Metadata` / `generateMetadata` / `ResolvingMetadata`** — Use the route `head` API; `loaderData` is typed from the `loader`. Set site-wide defaults on `__root`.
7. **`NextMiddleware` / `NextFetchEvent`** — Prefer `beforeLoad` on the root or leaf routes; true edge/request middleware stays in your host (Worker, proxy, etc.).
